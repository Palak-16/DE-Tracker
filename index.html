Tables, rows, and keys

A relational database stores data in tables (relations). Each row is one record; each column has a fixed type. Keys are how rows are identified and connected:

  • Primary key (PK) — uniquely identifies each row; can't be NULL; one per table. Can be one column (id) or several (composite key: order_id + product_id).
  • Candidate key — any column(s) that COULD be the PK (email is a candidate; you picked id). The rest become alternate keys.
  • Foreign key (FK) — a column holding another table's PK, creating the relationship. The DB can enforce it (referential integrity): you can't insert an order for a customer that doesn't exist.
  • Surrogate key — generated meaningless ID (auto-increment, UUID) vs natural key — real business value (email, PAN). Prefer surrogates; business values change.
-- constraints: rules the DB enforces for you
PRIMARY KEY   UNIQUE   NOT NULL   DEFAULT 0
CHECK (age >= 18)
FOREIGN KEY (customer_id) REFERENCES customers(id)

Relationships & cardinality

  • 1 : N (most common) — one customer, many orders. FK lives on the MANY side (orders.customer_id).
  • 1 : 1 — one user, one profile. FK + UNIQUE on either side (often just merged into one table).
  • M : N — students ↔ courses. Needs a junction table (enrollments: student_id + course_id), because you can't store many values in one FK column.

Normalization — with a worked example

Normalization = splitting tables to remove redundancy, so each fact is stored exactly once. Start with this mess:

orders_bad
order_id | customer_name | customer_city | products
1001     | Asha          | Pune          | "pen, notebook"
1002     | Asha          | Pune          | "pen"

1NF — atomic values: no lists inside a cell, no repeating groups. Every cell holds ONE value; every row is unique.

order_id | customer_name | customer_city | product
1001     | Asha          | Pune          | pen
1001     | Asha          | Pune          | notebook
1002     | Asha          | Pune          | pen

2NF — no partial dependency: (only matters with composite keys). Key here is (order_id, product). customer_name depends only on order_id — HALF the key. Split it out:

orders: order_id | customer_name | customer_city
order_items: order_id | product

3NF — no transitive dependency: non-key columns must depend on the key, nothing else. customer_city depends on the CUSTOMER, not the order → split again:

customers:   id | name | city
orders:      order_id | customer_id
order_items: order_id | product

The interview one-liner for 3NF: every non-key column depends on "the key, the whole key, and nothing but the key."

WHY IT MATTERS · Unnormalized, Asha's city is stored 3 times — updating one row and missing others creates contradicting data (an update anomaly). Normalization also prevents insert anomalies (can't add a customer with no order) and delete anomalies (deleting the last order erases the customer).

Denormalization is doing this in reverse, on purpose: analytics tables (your star schemas) merge dimensions back in or accept redundancy because reads with fewer joins matter more than write anomalies — the warehouse is rebuilt by pipelines anyway. OLTP → normalize; OLAP → denormalize deliberately. Higher forms (BCNF, 4NF) exist; 3NF is what jobs and interviews actually mean by "normalized."

Transactions & ACID

A transaction groups statements so they succeed or fail as one unit — the classic example is transferring money (debit + credit must BOTH happen).

BEGIN;
UPDATE accounts SET balance = balance - 500 WHERE id = 1;
UPDATE accounts SET balance = balance + 500 WHERE id = 2;
COMMIT;    -- or ROLLBACK; to undo everything
  • Atomicity — all or nothing; a crash mid-way rolls back.
  • Consistency — constraints hold before and after (no negative balance if CHECKed).
  • Isolation — concurrent transactions don't see each other's half-done work. Levels range from READ UNCOMMITTED to SERIALIZABLE (stricter = safer = slower).
  • Durability — once committed, it survives a crash (written to disk / WAL).

DE tie-in: this is exactly what plain files on S3 LACK, and what Delta/Iceberg's transaction log adds back on the lake.

Indexes — why queries are fast (or aren't)

An index is a sorted lookup structure (usually a B-tree) on chosen columns. Without one, WHERE email = 'x' scans every row (O(n)); with one, the DB jumps straight there (O(log n)) — like a book's index vs reading every page. PKs are indexed automatically.

  • Index columns you filter and join on frequently.
  • Trade-off: every INSERT/UPDATE must also update the index — over-indexing slows writes and eats space.
  • WHERE YEAR(created_at) = 2026 can't use the index on created_at — functions on the column defeat it. Rewrite as a range: created_at >= '2026-01-01' AND created_at < '2027-01-01'.
  • Composite index (a, b) helps queries filtering on a, or a+b — but NOT b alone (leftmost-prefix rule).

The SQL command families

DDL (structure):  CREATE, ALTER, DROP, TRUNCATE
DML (data):       SELECT, INSERT, UPDATE, DELETE
DCL (access):     GRANT, REVOKE
TCL (txns):       BEGIN/COMMIT/ROLLBACK
GOTCHA · DELETE removes rows one by one (can be rolled back, fires triggers); TRUNCATE resets the whole table instantly (DDL, usually no rollback); DROP deletes the table itself.

Views & other objects worth recognizing

View = a saved query that acts like a table (no storage; always fresh). Materialized view = the results physically stored and refreshed — faster reads, can be stale. This exact pair is dbt's view vs table materializations. A schema is a namespace grouping tables; a stored procedure is saved procedural code in the DB (rare in modern DE — dbt took its job).

Beyond relational — 30-second map

NoSQL families and when they fit: key-value (Redis — caching, sessions), document (MongoDB — flexible JSON records), wide-column (Cassandra — huge write-heavy workloads), graph (Neo4j — relationship-heavy queries). Relational remains the default when data is structured and consistency matters — which is most DE work.

Absolute basics — before your first drill

-- the skeleton (write in this order)
SELECT   name, COUNT(*) AS cnt      -- AS gives an alias
FROM     orders
WHERE    status = 'paid'            -- filters ROWS (before grouping)
GROUP BY name
HAVING   COUNT(*) > 3               -- filters GROUPS (after grouping)
ORDER BY cnt DESC                   -- DESC = biggest first
LIMIT    5;                         -- top 5 rows (LIMIT 5 OFFSET 10 skips 10)

But SQL executes in a different order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. That's why a column alias made in SELECT can't be used in WHERE — WHERE runs first.

-- filtering toolbox
WHERE age BETWEEN 18 AND 30      -- INCLUSIVE on BOTH ends (unlike Python range!)
WHERE city IN ('Pune', 'Delhi')
WHERE name LIKE 'A%'             -- % = any chars, _ = exactly one char
WHERE email IS NULL              -- never  = NULL
WHERE NOT (a AND b)              -- AND binds tighter than OR: use parentheses
-- strings use 'single quotes'; -- starts a comment
-- aggregates (they IGNORE NULLs, except COUNT(*))
COUNT(*)         -- all rows
COUNT(col)       -- rows where col is not NULL
COUNT(DISTINCT col)
SUM(x)  AVG(x)  MIN(x)  MAX(x)
SELECT DISTINCT city FROM users;    -- unique values
GOTCHA · Every column in SELECT must either be inside an aggregate (SUM, COUNT…) or listed in GROUP BY. Mixing them freely is the #1 beginner error.
GOTCHA · BETWEEN is inclusive of both ends — the opposite instinct from Python's range/slicing. Date BETWEEN '2026-01-01' AND '2026-01-31' includes both dates.

Joins — which one and why

INNER keeps only matches. LEFT keeps every left row (missing right side becomes NULL) — use whenever the question says "including those without…". Self join = same table twice with aliases (employee↔manager, today↔yesterday). CROSS = all combinations (calendar × products for filling gaps).

-- self join: compare a row to "yesterday's" row
SELECT w1.id FROM Weather w1
JOIN Weather w2 ON DATEDIFF(w1.recordDate, w2.recordDate) = 1
WHERE w1.temperature > w2.temperature;
TRAP · Filtering the right table in WHERE turns a LEFT JOIN into an INNER JOIN. Put the filter in the ON clause to keep unmatched rows.

NULL logic

NULL fails every comparison: x != 2 silently drops NULL rows. Use IS NULL / IS NOT NULL, COALESCE(x, fallback), and IFNULL. Aggregates ignore NULLs (COUNT(col) skips them; COUNT(*) doesn't).

TRAP · NOT IN (subquery) returns nothing if the subquery contains a single NULL. Use NOT EXISTS or filter NULLs inside the subquery.

Aggregation, HAVING, conditional counts

-- WHERE filters rows, HAVING filters groups
SELECT managerId FROM Employee
GROUP BY managerId HAVING COUNT(*) >= 5;

-- conditional aggregation: pivot logic without pivots
SUM(CASE WHEN state='approved' THEN amount ELSE 0 END)
AVG(action = 'confirmed')          -- avg of boolean = the RATE
COUNT(DISTINCT CASE WHEN x THEN id END)

The AVG(condition) trick answers most "percentage of…" questions in one line.

Window functions — the DE interview core

ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC)  -- 1,2,3,4
RANK()       -- 1,1,3 (gaps after ties)
DENSE_RANK() -- 1,1,2 (no gaps)  ← "top N per group" questions

LAG(x)  OVER (ORDER BY d)   -- previous row's value
LEAD(x) OVER (ORDER BY d)   -- next row's value

SUM(x) OVER (ORDER BY d)                             -- running total
AVG(x) OVER (ORDER BY d ROWS BETWEEN 6 PRECEDING
             AND CURRENT ROW)                        -- 7-day rolling

Pattern for "top N per group": window in a CTE, filter rn <= N outside — you can't put a window function in WHERE.

Dedup patterns

-- keep one row per key
WITH ranked AS (
  SELECT *, ROW_NUMBER() OVER (
    PARTITION BY email ORDER BY updated_at DESC) rn
  FROM users)
SELECT * FROM ranked WHERE rn = 1;

Gaps & islands (consecutive runs)

-- id - ROW_NUMBER() is constant within a consecutive run
SELECT *, id - ROW_NUMBER() OVER (ORDER BY id) AS grp
FROM t;   -- GROUP BY grp, keep groups HAVING COUNT(*) >= 3

Dates & strings (MySQL flavor)

DATE_FORMAT(d, '%Y-%m')   DATEDIFF(a, b)   DATE_ADD(d, INTERVAL 1 DAY)
CONCAT(a, b)   UPPER/LOWER   LEFT(s, n)   SUBSTRING(s, 2)   LENGTH(s)
GROUP_CONCAT(DISTINCT p ORDER BY p)        -- string agg per group
col REGEXP '^[A-Za-z][A-Za-z0-9._-]*@leetcode\\.com$'

CTE habits (dbt style)

One CTE per logical step, named after what it produces (daily_orders, ranked_users). Read top to bottom like a pipeline. If a CTE does two things, split it.

Quick trap list

  • Join fanout: joining 1-to-many then SUM → inflated numbers. Aggregate BEFORE joining.
  • Integer division: 3/2 = 1 in some engines — multiply by 1.0 first.
  • GROUP BY must include every non-aggregated select column.
  • ORDER BY tie-breakers: read the question — "smallest id if tie" is almost always there.
  • ROUND(x, 2) — LeetCode checks decimals exactly.

Core structures & complexity

list  : append O(1), pop() O(1), pop(0) O(n), x in list O(n)
dict  : get/set/in O(1)          set : add/in O(1)
d.get(k, default)    d.setdefault(k, [])    for k, v in d.items()

If you're checking membership in a loop, it should be a set. If you're counting, it should be a Counter. If you're grouping, it should be a defaultdict(list). Those three reflexes solve half of all drills.

Sliding window

seen, l, best = set(), 0, 0
for r, ch in enumerate(s):
    while ch in seen:            # shrink until valid
        seen.remove(s[l]); l += 1
    seen.add(ch)
    best = max(best, r - l + 1)
